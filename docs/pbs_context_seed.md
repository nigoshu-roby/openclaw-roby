# PBS Context Seed

このファイルは、Precision Sprint の次段階で PBS に人手で渡す前提知識をまとめる場所です。

目的:

- project / client / owner の理解を先に与える
- Gemini の大量投入前に、仕事の前提を人間の言葉で固定する
- task 抽出時の `wrong_project` / `not_actionable` / `owner違い` を減らす

使い方:

- まずは重要な project から順に埋める
- 全部を一気に書かなくて大丈夫
- 箇条書き中心で OK
- わからない欄は空欄で OK

---

## Task Definition

PBS における task とは、**特定の担当者が、一定期間内に、明確な行動として実行・判断・確認・返信・調整・作成すべき仕事の最小管理単位**です。

### 必須条件

- 行動可能である
  - 実際に何をするかが分かる
  - 動詞に落ちる
  - 実行・確認・返信・作成・調整・依頼・判断のいずれかに乗る
- 担当主体がある
  - 誰がやるかがある程度分かる
  - 少なくとも「自分が追うべきか」は判断できる
- 放置コストがある
  - 放置すると漏れ・遅延・判断停滞が起きる
  - task 管理ツールに載せる意味がある
- 完了判定ができる
  - 「終わった」と言える条件がある
  - 完了・未完了が曖昧すぎない

### task ではないもの

- 単なる情報
  - 背景説明、状況共有、会話メモ、雑談、感想
- 完了報告
  - すでに終わっていること、状態の報告だけで次アクションがないもの
- 論点だけの文
  - 問いはあるが、まだ行動に落ちていないもの
  - ただし「考える」「判断する」「提案する」が実務として必要なら task 化してよい
- 他人専用の仕事
  - 自分が追う必要のない他担当者 task

### 良い task の条件

- 短いが意味がある
- 何をするか分かる
- 文脈なしでもある程度読める
- 実務で着手しやすい
- Neuronic に置く意味がある

例:

- `見積書を送付する`
- `スマレジとの打ち合わせを調整する`
- `契約書を準備する`

### 悪い task の条件

- broad すぎる
- 何をするか分からない
- 報告と task が混ざる
- 文脈依存が強すぎる
- 完了条件が不明

例:

- `対応する`
- `確認`
- `スケジュール調整`
- `仮構築完了`
- `検討する`

### 運用上の最終定義

PBS における task とは、**自分または自分が追うべき担当主体に対して発生した、放置コストがあり、完了判定できる、次の具体的行動**とする。

### 判定ルール

- `task`
  - 行動あり
  - 担当あり
  - 放置コストあり
  - 完了判定あり
- `review`
  - 重要そうだが task 断定は弱い
- `drop`
  - 情報・報告・雑談・完了済み・他人専用

## 1. 自分の役割

- 名前: 新後周平
- 表示名の揺れ:
  - 例: `私`, `自分`, `にーご`, `shu`, `新後`
- 主な役割: 株式会社TOKIWAGIの従業員。テクニカルアーキテクチャー領域を担当。
- 主担当の仕事: クライアントの要望を受けて要件定義し、ベンダーへ相談すること。テストや仕様に合わせたSaaSツールの導入・設定、運用の土台に乗るまでの先導、ベンダーやツールベンダーへの交渉や相談も行う。クライアントとテクノロジーを結びつける翻訳者的な立ち回りを担う。ただし、ツール導入自体が目的ではなく、クライアントが実現したい目的を達成できる環境を構築することが主な目的。
- 主担当ではないが関わる仕事:
- 自分の task として上げてよい基準:
  - 自分が次に何か行動しないといけないもの
  - 自分が返信・確認・調整・作成・依頼する必要があるもの
  - 会議やメールの流れから見て、自分が担当者または窓口と読めるもの
  - 自分の担当案件に関する実務タスク
  - 期限や締切があり、自分が対応漏れすると困るもの
  - 自分が Neuronic 上で管理したほうが漏れ防止になるもの
  - プロジェクトの進行上、規則やルール、仕様に関わる重要な変更があるもの

- 自分の task に上げないでほしい基準:
  - 他担当者が実行する前提のもの
  - 自分が見るだけでよい共有事項
  - 雑談、状況報告、背景説明
  - まだ task にするには曖昧すぎる話
  - 会議メモをそのまま貼っただけのもの
  - 自分の担当範囲外の案件や作業
  - 他人の名前が明確に出ていて、その人が担当と読めるもの
  - 参考情報や保留中のアイデア

  - 他担当者が主担当だが、自分も状況把握しておきたいメール

- digest に留めてよいメール:
  - 参考情報
  - 定期通知
  - ツール通知
  - すぐに行動しなくてよい案内
  - あとでまとめて見ればよいメール

- archive してよいメール:
  - 明らかなメルマガ
  - 広告・営業色が強いメール
  - 自分の仕事に関係しない案内
  - no-reply の一般告知
  - すでに不要と判断できる自動通知

- task にしないでほしいメール:
  - 他担当者が実行すべき内容
  - 単なる雑談、共有、報告
  - まだ論点整理前で task として曖昧なもの
  - 一読だけで十分なもの
  - メルマガやサービスのお知らせ

- reply task にしてよい基準:
  - 明確に返信が必要
  - 質問への返答が必要
  - 日程回答が必要
  - 承認・可否返答が必要
  - 先方が自分からの返事待ちである

- action task にしてよい基準:
  - 資料作成
  - 内容確認
  - 関係者への確認
  - 見積・請求処理
  - 設定変更や依頼実行など、返信以外の実作業がある

## 5. 議事録 task 化ルール

- task にしてよい議事録メモ:
  - 自分が次に行動する必要があるもの
  - 自分が担当者または窓口と読めるもの
  - 誰が何をするかがある程度明確なもの
  - 期限、次回会議、依頼先、成果物などが見えるもの
  - 確認、調整、作成、送付、修正、依頼などの行動に落とせるもの

- task にしないでほしい議事録メモ:
  - 背景共有
  - 状況説明
  - 雑談
  - 課題感だけで、まだ action が決まっていないもの
  - 他人の担当作業
  - project に関係はあるが、自分の担当外のもの
  - 会議メモをそのまま貼っただけの文

- 自分 task にしてよい基準:
  - 自分がやるべき action が明確
  - 自分の担当案件である
  - 自分が調整役・確認役・返信役である
  - 自分が放置すると漏れになりやすいもの

- 自分 task にしないでほしい基準:
  - 高田さん、清さん、他担当者が主体のもの
  - クライアント側の宿題
  - ベンダー側の宿題
  - まだ誰の担当か決まっていないもの
  - 単なる議論メモやアイデア

- task 名の理想:
  - 短すぎず、何をするかが分かる
  - 会議メモの断片ではなく、行動文になっている
  - 可能なら「何を」「誰に/何について」が分かる
  - 例: 「見積を送付する」「日程候補を送る」「仕様確認を進める」

- subtasks に分けてほしい基準:
  - 1つの task に複数 action が混ざっている
  - 先に確認・製作、その後・報告・送付、のように順序がある
  - 1件の会議メモから複数の実行作業が出る

  - 「連絡先が変わりましたので、登録をお願いします」
- 望ましい判定:
  - task
- 望ましい task:
  - 「◯◯さんの連絡先を変更する」
- 元メールの内容:
  - 「打ち合わせをお願いします。」
- 望ましい判定:
  - task
- 望ましい task:
  - 「◯◯さんとの打ち合わせについて日程調整する」
  - 「◯◯さんとの打ち合わせについてアジェンダをまとめる」
- 元メールの内容:
  - 「不具合の確認をおねがいします。」
- 望ましい判定:
  - task
- 望ましい task:
  - 「◯◯についての不具合を確認する」
  - 「◯◯の不具合について、ベンダーに報告・調査依頼をする」
- 元メールの内容:
  - 「アカウント審査に失敗しました」
- 望ましい判定:
  - task
- 望ましい task:
  - 「◯◯のアカウントの再審査を実行する」

### 6.2 メールの task にしない例

- 元メールの内容:
  - サービスのアップデート案内
- 望ましい判定:
  - digest または archive

- 元メールの内容:
  - 背景共有だけで、返信や行動依頼がない
- 望ましい判定:
  - review
- 元メールの内容:
  - 他の担当者に向けたメールの内容
- 望ましい判定:
  - review

- 元メールの内容:
  - サービスの成功事例や失敗事例についての案内
- 望ましい判定:
  - review

- 元メールの内容:
  - LINE広告の予算消化案内
- 望ましい判定:
  - review
- 元メールの内容:
  - GASについての不具合報告
- 望ましい判定:
  - review

### 6.3 議事録の良い例

- 元メモ:
  - 「新後さんが見積を送る」
- 望ましい task:
  - 「見積を送付する」

- 元メモ:
  - 「来週までに高田さんに確認して先方へ返す」
- 望ましい task:
  - 親: 「確認結果を先方へ返す」
  - 子:
    - 「高田さんに確認する」
    - 「先方へ返答する」
- 元メモ:
  - 「先方はこの方針に不安がある」
- 望ましい task:
  - 「先方の不安材料を明確にする」

### 6.4 議事録の task にしない例

- 元メモ:
  - 「高田さんが実装を進める」
- 理由:
  - 他担当者 task

- 元メモ:
  - 「広告配信の方向性を検討」
- 理由:
  - まだ曖昧で broad すぎる

## 7. 優先して埋めてほしい項目

最初はここだけでも十分です。

- 自分の役割
- project の正式名 / 略称 / 主担当範囲
- 他担当としてよく出る人
- task にしてよいメール / review 止まりのメール
- task にしてよい議事録アクション / task にしない議事録メモ
